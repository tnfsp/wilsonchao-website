## 3. 資料合約（CC 照這個接）

### 你讀的檔（OWL 產出的公開子集）
- 路徑：`~/.openclaw/workspace/data/wilson-preferences.public.jsonl`（**OWL 會建好**，目前還沒建——見第 5 節 Q）
- 這是 raw 的**已審核、已去敏**子集。每筆是一行 JSON。
- CC 的 `sync-drawer.ts` 從這裡讀 → 轉成 `apps/main/content/drawer/*.json`（或單檔），跟現有 blog sync 同一個 pattern。

### 每筆欄位（公開版可能只露部分，以實際 public 檔為準）
```json
{
  "date": "2026-06-14",
  "questionId": "pq-20260614",
  "question": "想買個不便宜的東西時，你是哪派？研究魔人 vs 對眼就買",
  "optionA": "研究魔人，比價、看評測、爬一堆心得，確定了才下手",
  "optionB": "對眼就買，喜歡的感覺對了，當下就帶走",
  "choice": "偏B",
  "answer": "...",
  "reason": "...",
  "tags": ["消費", ...],
  "category": "money",
  "dimension": "消費"
}
```

### `choice` 五段制（渲染時可拿來做視覺刻度）
`A` / `偏A` / `混合` / `偏B` / `B` — 可以做成一條 A↔B 的刻度條，標出他落點。

### 紅線（不可違反）
- **CC 不直接讀 raw `wilson-preferences.jsonl`**，只讀 `.public.jsonl`。
- 公開檔若某筆某欄被 OWL 拿掉（例如 `reason`），就是故意的，別自己補回去。
- 完整隱私黑名單在 OWL 的 L1，但你不用管——你只會拿到已過濾的料。

## 5. 還沒決定 / 要跟 Wilson 對齊的事

動這些之前先問 Wilson，別自己定：

**A. 互動設計（Wilson 明說要先對齊再上）**
- **翻面**：碎片卡正面顯示 問題+choice，背面翻出 reason？還是 reason 直接顯示？
- **簽到本**：訪客留言/簽到的功能要不要？（Wilson 提過「簽到本」概念但沒定）
- **隨機抽**：隨機抽一張碎片給訪客看的互動？

**B. 一張碎片卡到底露多少**
- 只露 問題 + 他選哪邊（A/B 刻度）？
- 還是連 `reason` 一起露？（reason 是靈魂，但也最可能帶私人脈絡——這牽涉隱私，OWL 會在 public 檔層決定哪些 reason 能出，但**呈現方式**要 Wilson 點頭）

**C. 資料更新節奏**
- `sync-drawer` 手動跑，還是接進現有 CI（每日 deploy）？碎片一天長一筆，建議跟現有 vault sync 同一條 CI。

**D. 量體**
- 現在只有 4 筆，每天 +1。設計要同時撐得住「很少」和「很多」兩種狀態（空狀態 / 滿溢狀態都要好看）。

### Owl 的傾向（建議起點，非定案 — 最終以 Wilson 為準）

- **翻面：做。** 正面只放「問題 + 他落在 A↔B 哪裡」（choice 五段做成刻度條），背面才翻出 `reason`。理由：reason 是「為什麼」、是這個人真正所在的地方，藏在翻面後面 = 獎勵好奇、不喧嘩，也順手解掉「reason 露不露」——預設不吼出來，想懂他的人才翻。這跟抽屜的靈魂（把自己當一回事）一致。
- **reason：露，但只在翻面背後。** 資料層已去敏，留下來的是「私人但安全」，剛好適合「翻開才看到」。不要放正面。
- **簽到本：v1 先不做。** 它會把抽屜從「Wilson 的私密自畫像」變成「社交表演場」，稀釋親密感，還帶來審核/灌水負擔；而且現在才 4 張卡，空房間放簽到本很尷尬。真要互動，未來做輕版：訪客也選「這題你哪派 A/B」，看到自己站在 Wilson 旁邊——保持「認識自己」不變成「表演給人看」。但不是 v1。
- **隨機抽：設計時留位，先別上。** 主視圖該是「時間順序的紙條流」（按時間排 = 看著他一點點長出來）。隨機抽是錦上添花的「抽一張」彩蛋，碎片夠多（~15+ 筆）再加按鈕，現在 4 筆抽了沒意義。
- **更新節奏：接現有 CI**，每日 deploy 時一起 sync public 檔，別手動。碎片一天一筆，跟 vault sync 同一條管線最省事（自動化優先）。
